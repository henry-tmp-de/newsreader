### 1.1 用户画像构建

**实现位置**：[backend/src/main/java/com/newsreader/service/impl/RecommendationServiceImpl.java](backend/src/main/java/com/newsreader/service/impl/RecommendationServiceImpl.java) `buildUserProfile()`

#### 数据来源

用户画像从两张表中聚合数据：

| 数据维度 | 来源 | SQL 查询方法 |
|---|---|---|
| 已读文章数 | `learning_records`（`ARTICLE_COMPLETE`） | `getReadArticleCount()` |
| 阅读启动次数 | `learning_records`（`READ`） | `getCompletedArticleCount()` |
| 平均阅读时长（秒） | `learning_records`（`READ`，`duration`） | `getAvgReadDuration()` |
| 练习总量与正确率 | `learning_records`（`EXERCISE_DONE`，`correct`） | `getExerciseStats()` |
| 主动查词次数 | `learning_records`（`WORD_LOOKUP`） | `getLookupCount()` |
| 近14天活跃天数 | `learning_records`（`created_at`） | `getActiveDays14()` |
| 词汇本单词量 | `user_vocabulary` 表 | MyBatis-Plus `selectCount()` |

#### 能力分数计算（`estimateAbilityScore`）

在从数据库取到各统计量之后，计算一个 0~100 的综合能力分：

```
abilityScore = base(等级) + min(accuracy/5, 15) + min(vocabSize/40, 10)
             + readDurationBonus + min(activeDays14, 10)
```

各项说明：

- `base`：BEGINNER = 45，INTERMEDIATE = 60，ADVANCED = 75
- 练习正确率贡献：`accuracy / 5`，最多贡献 15 分（即正确率 75% 时满分）
- 词汇量贡献：每 40 个单词贡献 1 分，最多 10 分
- 阅读时长奖励：平均时长 ≥ 150s 加 6 分，≥ 90s 加 3 分
- 近14天活跃天数：每活跃一天加 1 分，最多 10 分
- 最终值钳制在 [20, 95] 之间，防止极端值

#### 目标难度推断（`estimateTargetDifficulty`），i+1 原则实现

根据等级和能力分，推断用户当前应匹配的文章难度（EASY/MEDIUM/HARD）：

```java
if (level == ADVANCED)     → abilityScore >= 82 ? HARD : MEDIUM
if (level == INTERMEDIATE) → abilityScore >= 78 ? HARD : MEDIUM
// BEGINNER:
                           → abilityScore >= 58 ? MEDIUM : EASY
```

这体现了 Krashen 的"i+1"可理解输入假说：用户的当前水平是 i，推荐的目标难度始终比其当前等级略高一级，不推太难（挫败感）也不推太简单（无成长）。

### 1.2 文章难度计算

**实现位置**：`RecommendationServiceImpl.resolveDifficulty()`

文章的 `difficulty` 字段由两种来源填充：

1. **LLM 标注**（优先）：文章入库时异步调用 DeepSeek，`AIServiceImpl.assessDifficulty()` 让模型判断 EASY/MEDIUM/HARD（见 [AIServiceImpl.java](backend/src/main/java/com/newsreader/service/impl/AIServiceImpl.java)）。
2. **文本可读性回退计算**（LLM 未标注时）：通过以下可读性指标计算：
   - `sentenceLen = 总词数 / 句子数`（句子以 `.!?` 分割）
   - `rareRatio = 长度 ≥ 9 字符的单词数 / 总词数`（近似学术词/低频词比例）
   - 规则：`sentenceLen > 24 或 rareRatio > 0.18` → HARD；`sentenceLen < 12 且 rareRatio < 0.07` → EASY；其余 → MEDIUM

### 1.3 推荐打分算法（多因子线性加权）

**实现位置**：`RecommendationServiceImpl.scoreArticle()` 及各子评分方法

候选文章池来自数据库最新 200 篇（`ORDER BY published_at DESC`），对每篇文章计算总分（0~100 分钳制），主要由以下因子构成：

#### 因子一：难度匹配分（`difficultyMatchScore`）

将 EASY/MEDIUM/HARD 映射为整数等级 1/2/3，计算与目标难度的差距：

| 差距 | 得分 |
|---|---|
| 0（完全匹配） | 28 |
| 1（相差一级） | 16 |
| 2（跨两级） | 4 |

这是权重最高的因子，确保难度契合的文章优先被推荐。

#### 因子二：兴趣匹配分（`interestMatchScore`）

用户的兴趣标签存储为逗号分隔字符串（`user.interests`），被解析为 `Set<String>`，然后：

1. 若文章的 `category` 字段与兴趣 Set 中任一项完全匹配 → +20 分
2. 对每个兴趣标签，如果出现在文章 `title` 或 `keywords` 中 → +8 分/次
3. 上限钳制为 35 分
4. 无兴趣标签时赋予基础分 8 分（保底）

#### 因子三：反馈偏好分（`feedbackPreferenceScore`）

从历史 LIKE/DISLIKE 记录中，按分类统计净偏好分：

```sql
SELECT a.category,
       SUM(CASE WHEN lr.action_type = 'ARTICLE_LIKE' THEN 1 ELSE -1 END) AS score
FROM learning_records lr JOIN articles a ON lr.article_id = a.id
WHERE lr.user_id = #{userId}
  AND lr.action_type IN ('ARTICLE_LIKE', 'ARTICLE_DISLIKE')
GROUP BY a.category
```

每个分类的净分（LIKE 次数 − DISLIKE 次数）再线性映射为推荐增减分：

```
raw > 0 → +min(raw * 4, 16)  （每次 LIKE 贡献 4 分，最多 +16）
raw < 0 → +max(raw * 4, -16) （每次 DISLIKE 扣 4 分，最多 -16）
```

#### 因子四：时效性分（`freshnessScore`）

```
score = max(0, 22 - 发布天数 × 2)
```

文章越新得分越高，11 天以上的文章时效分为 0。发布时间未知时赋予基础分 8 分。

#### 因子五：信息密度分（`infoDensityBonus`）

与难度计算共用同一套可读性指标，筛选信息密度适中的文章（排除过短或信息量异常的文章）：

- 基础分 6 分
- `0.08 ≤ rareRatio ≤ 0.18`（适中的学术词密度）→ +5 分
- `14 ≤ sentenceLen ≤ 24`（适中句长）→ +5 分
- 上限 16 分

#### 最终排序

```
总分 = 难度匹配(≤28) + 兴趣(≤35) + 反馈偏好(±16) + 时效(≤22) + 密度(≤16)
```

全部候选按总分降序排列，总分相同时按发布时间降序，取前 `size`（默认 12）篇。

### 1.4 用户反馈闭环

**实现位置**：`RecommendationController.feedback()` + `RecommendationServiceImpl.recordArticleFeedback()`

#### 反馈入库逻辑

反馈不做叠加，而是**以 articleId 为粒度做 Upsert**：

```java
// 删除该文章所有历史反馈（防止同时存在 LIKE 和 DISLIKE）
learningRecordMapper.delete(... userId + articleId + IN('ARTICLE_LIKE','ARTICLE_DISLIKE'));
// 插入新反馈
learningRecordMapper.insert(new LearningRecord(userId, articleId, 'ARTICLE_LIKE'/'ARTICLE_DISLIKE'));
```

这样同一篇文章对同一用户永远只有一条最新反馈，避免矛盾信号并节省存储。

#### 不感兴趣文章过滤

`recommendArticles()` 在构建候选列表时，会先查出所有 DISLIKE 记录的文章 ID 集合，在遍历候选池时跳过：

```java
Set<Long> disliked = new LinkedHashSet<>(learningRecordMapper.getDislikedArticleIds(userId));
// 循环中:
if (disliked.contains(article.getId())) continue;
```

同时，前端也在收到 DISLIKE 反馈响应后立即从本地列表中移除该卡片（乐观更新），用户无需刷新即可感受到效果。

### 2.1 词句即时释义

**实现位置**：[LearningServiceImpl.java](backend/src/main/java/com/newsreader/service/impl/LearningServiceImpl.java) `lookupText()` + [AIServiceImpl.java](backend/src/main/java/com/newsreader/service/impl/AIServiceImpl.java) `lookupText()`

#### 调用 LLM 的 Prompt 设计

```
Text: "<用户选中的词/句>"
Context: "<所在段落文本>"
Type: word / sentence
Return JSON with keys definition, chinese, example.
If type is sentence, definition should explain the sentence meaning in English and example can be empty.
```

LLM 返回 JSON 字符串，后端用 Jackson 解析为 `{definition, chinese, example}` 三字段 Map 发往前端。遇到 LLM 返回 Markdown 代码块包裹的 JSON 时，先清洗 ` ```json ` 标记再解析。

#### 查词行为的双重沉淀

`lookupText()` 在返回释义的同时，将查词结果写入 `user_vocabulary` 表：

- 若用户**首次**查该词 → INSERT 新词条，`mastery_level = 0`，`next_review_at = 当前时间 + 1天（单词）/ + 2天（句子）`
- 若该词**已存在** → 仅插入 `WORD_LOOKUP` 类型的学习记录（用于行为统计），不重复入词库

这样既完成了词汇的自动管理，又保证了行为数据的完整性，两条链路互不干扰。

### 2.2 文章 AI Copilot 问答

**实现位置**：`LearningServiceImpl.chatWithArticle()` + `AIServiceImpl.chatAboutArticle()`

#### 实现原理

每次问答时，后端将**文章全文（截取前 3500 字符）**拼入 Prompt，确保 LLM 的回答始终基于当前文章内容而非通用知识库：

```
Article content: <articleContent[:3500]>

Conversation history:
<历史对话记录（role: content 格式逐行拼接）>

User question: <当前问题>
```

System Prompt 为：`"You are an article reading copilot. Answer only based on the article. If unclear, say what is missing."` —— 约束模型仅在文章范围内回答，避免幻觉式发挥。

历史对话通过前端传入的 `history` 数组（`[{role, content}]`）拼接为文本追加到 Prompt，实现多轮连续对话的上下文保持。每次问答行为也会写入 `learning_records`（`action_type = 'ARTICLE_CHAT'`），记录用户的学习深度。

### 2.3 文章摘要的异步生成

**实现位置**：`AIServiceImpl.generateSummary()`

文章入库时会异步触发 LLM 生成 2~3 句的英文摘要：

```
System: "You are an English learning assistant. Generate a concise 2-3 sentence summary of the article."
User: <articleContent[:2000]>
```

摘要写入 `article.summary` 字段，用户进入阅读页时可直接获取，无需等待。若 LLM 未成功生成，回退为文章内容前 200 字。

### 3.1 题目动态生成

**实现位置**：[ExerciseServiceImpl.java](backend/src/main/java/com/newsreader/service/impl/ExerciseServiceImpl.java) `generateForArticle()` + `AIServiceImpl.generateExercises()`

#### Prompt 设计

```
Article:
<articleContent[:2000]>

Generate 5 English learning exercises for a <BEGINNER/INTERMEDIATE/ADVANCED> level student.
Include a mix of vocabulary, comprehension, and grammar questions.
Return JSON array:
[{
  "type": "COMPREHENSION",
  "question": "...",
  "options": ["A. ...", "B. ...", "C. ...", "D. ..."],
  "correctAnswer": "A",
  "explanation": "..."
}]
```

设计要点：

1. **上下文注入**：文章正文截取前 2000 字符注入 Prompt，让题目真正基于当前文章，而非通用题库
2. **等级自适应**：`userLevel` 参数传入 Prompt，让 LLM 生成与用户能力匹配的词汇难度
3. **题型混合指令**：`a mix of vocabulary, comprehension, and grammar questions`，一次生成三类题型
4. **结构化 JSON 输出**：规定 LLM 返回固定格式 JSON，后端解析后直接入库

#### 结果解析与入库

LLM 返回的 JSON 字符串在 `AIServiceImpl` 中先清洗 Markdown 代码块标记，再用 Jackson 解析为 `JsonNode` 数组，逐条映射为 `Exercise` 实体入库。生成的题目**会被缓存**：下次同一用户请求同一文章的练习时，`getByArticleId()` 直接查数据库返回，无需重复调用 LLM（降低 API 成本）。

### 3.2 答案提交与自动评估

**实现位置**：`ExerciseServiceImpl.submitAnswer()`

判题逻辑为字符串不区分大小写比较（`correctAnswer.equalsIgnoreCase(userAnswer.trim())`），结果写入 `learning_records`：

```
action_type = 'EXERCISE_DONE'
correct     = true / false
score       = 10 / 0
```

答题**错误时**调用 LLM 提供原因解析：

```
Question: <题目>
User's answer: <用户答案>
Correct answer: <正确答案>
Context: <题目解析文本>
Provide a brief explanation (2-3 sentences) of why the correct answer is right, in English.
```

用户不只知道"答错了"，还能理解"为什么这个选项是对的"，形成完整的学习闭环。

答题行为中的正确率数据会在下次构建用户画像时参与能力分计算，进而影响推荐文章的难度目标，形成"练习 → 画像 → 推荐"的三模块联动。

## 三大模块的数据流打通

三个模块并非独立的，它们共享同一张 `learning_records` 行为表，形成完整的学习数据闭环：

```
文章阅读（READ）
词句查询（WORD_LOOKUP）      →  learning_records  →  用户画像构建
练习作答（EXERCISE_DONE）                               ↓
文章 AI 对话（ARTICLE_CHAT）                     能力分、目标难度
反馈标记（ARTICLE_LIKE/DISLIKE）                        ↓
                                               推荐打分（5个因子）
                                                        ↓
                                               智能推荐文章列表
```

用户在阅读、练习、对话中产生的每一条行为都会反哺到推荐系统，使推荐结果随着用户的持续使用而逐渐精准。

## 后续增强建议（可选）

### 推荐模块

- **反馈时效衰减**：在 `getCategoryPreferenceScores` 查询中引入时间权重，近期反馈权重更高：
  ```sql
  SUM(... * CASE WHEN lr.created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY) THEN 2 ELSE 1 END)
  ```
- **不感兴趣原因细化**：在反馈接口中增加 `reason` 字段（`TOO_HARD/TOO_LONG/NOT_RELEVANT`），存入 `learning_records.note`，用于更精细的调整（如答题正确率持续偏低时自动降低目标难度等级）

### 练习模块

- **考试题型结构化**：在 `generateExercises` Prompt 中加入格式模板，支持完形填空、长篇匹配等 CET-6/考研/IELTS 专项题型
- **答案质量校验**：解析 LLM 返回的题目时增加校验（选项不足4个、答案不在选项中等情况自动过滤或重试）

### 阅读辅助模块

- **词汇掌握度自适应复习**：基于 `mastery_level` 和 `next_review_at` 字段，在首页或词汇本中推出间隔重复（Spaced Repetition）复习队列
- **段落级摘要**：对长文生成分段摘要，支持用户快速定位感兴趣的段落
